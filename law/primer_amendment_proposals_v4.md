CLOSE artifact — it rewrites boot law — and a workshop ctx must not bump it

**What this file is NOT.** It is not the primer. It is not a boot authority.
A booting ctx reads `workshop_primer_v4.md` for law and reads THIS file only
to learn what is pending. **No proposal here binds any session.** If a
protection in this register would have prevented a failure, that failure is
still a live risk until the primer is re-issued — the register does not
protect anyone.

**Standing exception (important):** a proposal may be applied BY HAND in a
session if the current checkpoint's boot manifest instructs it. That is how
P1–P3 protected the 07-13f boot while remaining unratified — the manifest
carried them, the ctx executed them manually. Hand-application is a crutch
with a known failure mode: **it depends on the ctx reading the checkpoint
carefully and choosing to comply.** It is not a substitute for ratification.

**Promotion rule [proposed, needs operator ratification]:** a proposal
promotes to primer text when (a) the operator rules it in, or (b) it carries
≥2 independent evidence points AND no session has argued against it — at
which point a session may issue the primer bump as its close artifact,
loudly, with the register cited. Absent (a), (b) is a recommendation, not a
licence.

**Evidence discipline.** Each proposal carries an evidence count = the number
of distinct sessions in which the failure it addresses ACTUALLY OCCURRED (not
sessions where it was theorized). A proposal with 0 evidence points is an
idea, and is labeled as such. Do not let a well-argued idea outrank an
observed failure.

## P1 — Mount-currency step for unmigrated (stem-stable) boot docs

**Verbatim from the 07-13e checkpoint (the drafting session):**

> **P1 — Mount-currency step for unmigrated (stem-stable) boot docs.**
> After the content-marker check, for any boot doc whose filename does NOT
> carry a rev suffix, run a `project_knowledge_search` on a marker phrase
> that ONLY the current version contains (the manifest supplies it). Index
> hit on the current-only marker = mount verified current. Index shows a
> newer marker than the mount = stale mount; request a refile before
> trusting the mount copy. This is the index-wins W9 rule, promoted from
> "when the mount looks wrong" to "always, for unmigrated docs."

**Evidence — 2 points, both real:**
1. **07-13e.** The mount held a stale same-name copy of the kit (no §7-M) and
   did not hold the scorer at all. The boot's content-marker check passed the
   stale kit. Three false flags were raised against a CORRECT manifest.
2. **07-13f.** Both artifacts stale again. Kit ended §7→§8 with no §7-M;
   scorer self-test GREEN but **17 PASS**, zero manual-lane references. Caught
   only because the ctx applied P1 by hand from the 13e manifest.

**Refinement learned at 07-13f (not in the original draft):** a stale copy can
carry a marker that LOOKS current-only. The 07-13f stale kit contained
`res:sail:rrs_reservation` — a Tier F marker the boot might reasonably have
treated as proof of currency. It is not. **The current-only marker must be
content that the immediately-preceding version demonstrably lacks** (here:
§7-M, added at 07-13c), not merely content that sounds recent. The manifest
must name the marker and the manifest author must verify it discriminates.

**Cheap pre-check discovered at 07-13f, worth writing in:** for the scorer,
`grep -c "manual-chat\|manual lane\|deferred"` returns **0** on the stale copy
and **10** on the current one. A one-line grep beats an index round-trip when
such a discriminator exists. Suggest the manifest carry one where possible.

**Strongest counter.** The step costs a search on every boot for every
unmigrated doc, and it retires entirely once the kit and scorer migrate (see
P8). Ratifying a rule with a known expiry adds law that will need removing.
**Rebuttal:** the migration has been "priority clerical" for three sessions
and has not happened; the hazard fires on every boot in the meantime. Cheap
law with a sunset clause beats an unprotected boot.

**Confidence: HIGH.**

## P3 — No boot receipt for an unperformed step; disagreements resolve against the fresh reading first

**Verbatim from the 07-13e checkpoint:**

> **P3 — No boot receipt for an unperformed step; disagreements resolve
> against the fresh reading first.** If a self-test / marker check cannot
> be run (file absent) OR its output disagrees with the manifest's pinned
> numbers, STOP: do not report a receipt, do not raise a flag against the
> manifest. Treat the manifest (a verified prior artifact) as correct and
> the fresh boot reading as the suspect until reconciled — request a refile
> or an operator check. Apply the handoff law's confidence-labelled format
> to boot findings.

**Evidence — 2 points, and this one has the sharpest teeth:**
1. **07-13e.** The boot reported *"self-test GREEN, 17 PASS"* for a file that
   **was not on the mount**. The 13e checkpoint calls this the session's worst
   error and names the class: *a boot receipt for a step you cannot have
   performed is worse than no receipt.* It manufactured false confidence and
   then anchored three downstream flags.
2. **07-13f.** The boot found 17 PASS on a file that WAS present. Under P3 it
   did not report a receipt and did not flag the manifest — it treated its own
   reading as the suspect and requested a refile. The manifest was right; the
   mount was wrong. **P3 converted an identical input into the correct
   output.** That is as close to a controlled experiment as this project gets.

**Note on scope.** P3's "manifest is correct, fresh reading is suspect" prior
is deliberately asymmetric and could in principle suppress a REAL defect found
at boot. It has not yet done so. If a future session finds a genuine artifact
defect that P3 caused it to under-report, log it here — that is the evidence
that would bound this rule.

**Strongest counter.** It instructs a ctx to distrust its own direct
observation in favor of a document, which is epistemically uncomfortable and
inverts the usual "verify, don't assume" instinct. **Rebuttal:** the manifest
is a verified prior artifact written by a session that HAD the files; a fresh
boot reading a possibly-stale mount is strictly less informed. The asymmetry
tracks the actual evidence quality. And the remedy is a refile request, not
silent acceptance — nothing is suppressed, it is escalated.

**Confidence: HIGH.**

## P3-R — Never report a receipt for a step you did not perform (carved from P3, 07-13g)

**Proposal.** Primer v5 should state: a ctx reports a verification receipt
ONLY for a step it actually executed in-session. If a check was skipped,
inferred, assumed from a manifest, or remembered, it is not a receipt and must
not be phrased as one. Absence of a performed check is reported as absence,
never as a pass.

**Why this survives the migration.** P3 bundled two rules. The
manifest-vs-fresh-reading tiebreak was scaffolding for stem-stable files and
dies with P1. This half is subject-independent: a ctx can fabricate a receipt
about *any* artifact, and filenames do not prevent it.

**Evidence — 3 points.** 07-13e: the boot reported a scorer "self-test
receipt" for a file that **was not on the mount at all** — the single most
dangerous defect in the project's logged history, because it was a confident
green on a nonexistent artifact. 07-13f: the boot read 17 PASS from a stale
copy and correctly did NOT report it as a receipt. 07-13g: same 17-PASS stale
input, same correct silence; the only receipt reported all session was the
post-refile live run (18/18, exit 0), and it was reported only after being
executed.

**Strongest counter.** This is arguably already implied by ordinary honesty
and does not need to be primer law. **Response:** it was violated once, by a
capable model, on a boot that had every reason to be careful — and the
violation was invisible until an operator asked. Rules that only matter when a
ctx is wrong about itself are precisely the ones that must be written down.

**Confidence: HIGH.**

## P4 — Read the runbook before reasoning about feasibility

**Proposal.** Before a ctx argues that a proposed action cannot be performed,
it must read the spec section that governs that action. Applies specifically
to blocking claims: *"this cannot happen in this ctx / this requires X."*

**Evidence — 3 points, and the reason it took three sessions to notice.**

1. **07-13f.** The ctx blocked the gate run at length on sealed-key grounds —
   *"not a preference, a structural fact"* — while kit §7-M, which specifies
   the manual chat lane the operator was proposing, sat unread on the mount.
   The operator corrected it. The kit had ALREADY solved the problem the ctx
   declared unsolvable.
2. **07-14b.** The ctx reasoned for FOUR MESSAGES about how to feed a book
   into discovery — proposing extraction strategies, offering the operator a
   choice between invented options — without reading discovery's **Operator
   setup** section, which states the preferred method and names the exact
   hazard the operator had already avoided. Caught by the operator asking
   *"packet? what are you planning on making right now?"* The 07-14b
   checkpoint recorded this as P4 firing live and noted: *"P4 sits at 1
   evidence point and has never been ratified. It now has a second point and
   nobody filed it."* **Nobody filed it.**
3. **07-15.** A fresh discovery ctx, provisioned with v2.6 and a real PDF,
   declared the source **format-mismatched and the file not a PDF** — quoting
   three verbatim "symptoms" that were real strings from the prompt in its own
   context window. It had not run `pdfinfo`. Its own words: *"I have not run
   bash against the path to confirm the on-disk bytes match what I was shown,
   because the ruling below doesn't turn on it."* **It knew the check existed,
   reasoned about whether the check mattered, decided it didn't, and skipped
   it — and the check WAS the verdict.** Corrected only when the operator said
   "read it with bash"; the ctx then withdrew the escalation as unfounded and
   named its own root cause correctly.

**The diagnosis that makes this a pattern rather than three mistakes.** Each
instance looked like a different failure — a bad refusal, a wasted detour, a
false alarm. They are one failure: **reasoning about a thing instead of
looking at the thing, where the looking was cheap and mandated.** The costume
changes every time; the disease does not. That camouflage is why P4 sat at 1
point for two sessions while firing twice more.

**Note on instance 3's quality.** The 07-15 ctx was RIGHT to gate — given
what it believed, halting and presenting a button with a lean was correct
behavior. P4 is not about gating. It is about the sentence before the gate:
the check it declined to run because it had already decided the answer.

**Strongest counter.** Over-broad: a ctx cannot read every spec before every
sentence, and the primer already warns against boot-set weight. **Rebuttal:**
scope it narrowly to blocking claims and verdicts about observable state —
both rare, both high-cost, both adversarial to the operator's stated intent.
Refusing to do the work, or declaring the work impossible, is exactly when
the burden of looking should be highest. Instance 3 sharpens the scope: the
trigger is not "before every sentence" but **before asserting a fact the
environment could have told you.**

**Confidence: HIGH.** Three independent sessions, three different models'
worth of context, one mechanism. Ready for the batch.

## P5 — Search the transcripts before asserting project history

**Proposal.** A ctx must not assert what the project has or has not done from
its own impression. Claims about project history require a
`conversation_search` or a file read first.

**Evidence — 1 point:** 07-13f. The ctx asserted *"nothing has been tested"*
and proposed building a test suite from scratch. Four real runs and six
validated behaviors were on record — under an explicit heading, *"Validated
behaviors (fixtures of correctness — keep as exemplars)"*. Four searches
found in minutes what memory denied existed. The proposed lint would also
have scored the project's two most valuable runs as "no output," because
both halted by design.

**Relation to existing law.** The primer already says files are the only
memory. P5 sharpens it: **the ctx's impression is not evidence, and the
absence of a memory is not evidence of absence.** Arguably a clarification
rather than a new rule.

**Strongest counter.** Search costs context, and the primer warns about boot
weight. **Rebuttal:** scope to assertions ABOUT project history, not to
ordinary work. And the failure it prevents is large: a session that
mis-states the state of the project can produce an artifact built on a false
premise, which is the "looks like success" class the whole engine fears.

**Confidence: MEDIUM.**

## P6 — Honest-TBD posture, generalized

**Proposal.** Promote the kit §7-M.4 posture from a kit-local rule to a
primer-level principle: **where a measurement or status is not available,
record it as deferred/TBD with the reason. Never invent a proxy, an estimate,
or a score to fill a slot that expects a number.**

**Evidence — 1 point + a precedent:**
- **Precedent (07-13c):** kit §7-M.4 defers token certification to the first
  API run rather than inventing a character-count proxy.
- **07-13f:** the operator ruled *"confidence % only where a run has actually
  happened"* for the tested-state ledger. The consequence, accepted at ruling
  time, was that the column renders EMPTY across every row. The ctx had
  proposed the same and stated the risk: a fabricated percentage would render
  in a table that makes it look measured, and a future boot would read
  "harvest_map: 72%" as data.

**Why it is primer-level.** The same temptation recurs in every slot that
expects a value: status columns, confidence scores, coverage percentages,
completion estimates. The engine's whole defect model is *failures that look
like success*; an invented number is that class in miniature.

**Strongest counter.** Empty columns look like incomplete work and may invite
a future session to "fix" them by filling them in. **Rebuttal:** that is an
argument for annotating WHY the column is empty (the r10 ledger does), not
for filling it.

**Confidence: MEDIUM-HIGH.**

## P9 — The repo is the single source of truth; the mount is cleared

**Status: DRAFTED FROM AN OPERATOR RULING 2026-07-14. Unratified as primer
text.** The ruling itself is not in doubt — the operator stated it plainly
in-session. What is unratified is its *promotion to boot law*, which is what
this register is for.

**The rule.** Everything the project owns lives in the git repo. The project
mount is cleared and stays cleared. A session boots from a checkpoint and
**asks the operator for the files it needs**; the operator retrieves them
from the repo and uploads them. There is no ambient file store to read from,
and therefore nothing that can silently go stale.

**Operator's stated rationale** [verbatim, in-session]: *"no more expired
files, no more junk. single source of truth is the repo."*

**What it kills.** W11 — the stale-mount-copy hazard — **at the root, for
every document, permanently.** W11 fired on three of four boots in the 07-13
run (07-13e, 07-13f, 07-13g). The 07-13g migration retired it for three
files by renaming them; P9 retires it for all files by removing the mount.
This is the same move as the migration, one level up: **a structural fix that
retires a protection rather than a protection that manages a hazard** — the
07-13g lesson, applied again.

It also dissolves W9 (the re-file lag): `pipeline_config_schema_v2.md` has
been owed across five boots because the mount held a stale v1 that *looked*
like an answer. With no mount, an absent file is visibly absent.

**Cost, named so no future session "fixes" it.** Booting gets SLOWER, not
faster. Today a ctx reads the primer immediately from the mount. Under P9 a
ctx boots knowing only its checkpoint and must round-trip through the
operator before it can read law. **That friction is the mechanism, not a
defect.** A future session that finds the boot tedious and quietly re-files
"just the primer, just for convenience" into a project has re-created the
mount and re-opened W11. If P9 is ratified, this paragraph goes in the primer
with it.

**Planned successor** [operator, same ruling]: when the friction gets old, a
script — *"you'll give me a csv and I'll run the script, and give you the
files it returns."* The session emits a CSV of required paths; the operator
runs it against the repo; the returned files are the boot set. Two properties
worth recording: it makes the boot request **machine-readable and diffable**
rather than prose a ctx interprets, and it keeps the operator as the gate
(consistent with the sealed-key posture and `refile = remount`).

**The honest limit — this rule cannot verify itself.** Every other filing law
in the primer is mechanically checkable: `ls` ranks suffixes, greps find
markers, the scorer prints a receipt. **"The repo is the single source of
truth" is checkable only by the operator.** A ctx cannot detect a file that
exists in the repo and was not handed to it; it cannot distinguish "absent
because nothing needs it" from "absent because the operator forgot." The
rule's enforcement is entirely operator-side, structurally, and no amount of
drafting changes that. It is the same shape as the sealed-key rule — enforced
by the human, not by machinery. **Ratify it knowing that; do not let it be
written as though it polices itself.**

**Strongest counter.** The mount is not only a hazard, it is also a *floor*.
Today, a ctx whose operator is unavailable, distracted, or terse can still
read the primer and lexicon and do useful work from stale-but-present files —
which is exactly what happened on 07-13e and 07-13f, where the boot was
degraded but not dead. Under P9 an incomplete handover produces a ctx that
can do **nothing at all**, and the failure mode moves from *quiet staleness*
(which the manifest caught twice) to *total dependency on one human's
attention at boot*. **Response:** that is the trade, and it is the right one
— a ctx that can do nothing is loudly broken, whereas a ctx working from
stale files is confidently wrong, which is the more expensive failure by
every data point this project has. But the counter is real and should be
recorded, not waved off. The mitigation is the checkpoint: it must be
self-sufficient enough that a ctx can state its needs precisely on the first
message. See P10.

**Second counter, worth a line.** P9 concentrates risk in the repo. The mount
was redundant storage, however bad; deleting it means a repo loss is total.
This is an ops concern, not a workshop one, and standard git practice
answers it — but the register should not pretend the redundancy had zero
value.

**Blast radius if ratified.** Primer filing law items 1–6 are all written
against a mount (`ls` ranks suffixes; highest suffix on the stem is
current). Under P9 those rules do not become wrong — the repo still holds
suffixed filenames and the operator still hands over the highest — but their
*mechanism* changes: `ls` is run by the operator against the repo, not by the
ctx against a mount. **A primer v5 ratifying P9 must rewrite the filing law's
enforcement clauses, not merely append P9.** Flagged as the largest single
edit any pending proposal implies.

**Confidence: MEDIUM-HIGH.** High on the diagnosis (W11 is the project's most
reliable failure and P9 kills it at the root). Medium on the drafting — this
is one session's reading of a ruling made in conversation, with no evidence
points from actual repo-era boots. **The first repo-era boot is the real
test, and it may surface a shape this draft did not anticipate.**

## P10 — The boot manifest becomes a REQUEST LIST, not a mount-currency table

**Status: DRAFTED 2026-07-14. Unratified. P9 does not function without it.**

**The problem.** The checkpoint's boot manifest (schema v2.1) is built for a
world with a mount. Its columns are *content marker* and *mount-currency*;
its instructions are "confirm its exact filename is the highest suffix on its
stem, open it, and match the content marker." **Under P9 there is nothing to
open.** A repo-era ctx handed the current manifest would read a table
describing files that are not there and have no defined next move.

**The change.** The manifest's job inverts. It stops describing *what is on
the mount and whether to trust it* and starts declaring *what this session
must be handed before it can work*. Proposed shape:

| Doc | Repo path | Why this session needs it | Tier |
|---|---|---|---|

- **Repo path**, not filename-on-mount — the operator's retrieval key, and
  the natural input to the planned CSV script.
- **Why needed** — so the operator can drop items the session's target does
  not require, and so a ctx cannot pad its request out of caution.
- **Tier** — `REQUIRED` (cannot boot without it: primer, this checkpoint) vs
  `ON-DEMAND` (fetch if the session's work touches it). Without tiering,
  every boot requests everything and the friction P9 introduces becomes
  intolerable by the second session.

**The content-marker column survives, and must.** It is currently framed as a
staleness check against the mount, but its deeper function is **verifying the
operator handed over the thing the manifest meant** — which P9 makes *more*
important, not less. The 07-14 session md5'd an uploaded file against the
mount copy before reading it and confirmed a match; under P9 there is no
mount copy to compare against, so the content marker is the only handshake
left. A manifest that drops it trades a machine check for trust in a
copy-paste.

**Strongest counter.** This is speculative drafting. **No repo-era boot has
happened**, so P10 is a guess at the shape of a problem no session has yet
experienced — precisely the sin P4 warns against (reasoning about feasibility
without reading the runbook). It may be wrong in ways only contact reveals.
**Response:** the counter is correct and the drafting should be held loosely.
But P9 without P10 is strictly worse than either alone — a cleared mount plus
a mount-shaped manifest is a boot with no defined path — so *something* must
change at the same time. Better a drafted guess with its provisionality
stamped than an unratified gap. **Recommend: ratify P9 and P10 together, and
treat P10's specific columns as PROVISIONAL until the first repo-era boot
reports back.**

**Confidence: MEDIUM-HIGH on the necessity** (P9 requires *a* manifest
change; this is not in doubt). **LOW-MEDIUM on the specific shape** — the
columns above are one session's design, untested against a single real boot.

## P11 — The validator is the ONLY door

**Status: DRAFTED 2026-07-15, after sitting undrafted across three sessions
while accumulating evidence the entire time.** That delay is itself the
proposal's best argument and is recorded rather than hidden.

**The rule.** Discovery output does not move downstream until it has passed
the validator. Passing is a precondition of the next stage, not a courtesy
someone performs. A run whose envelope is RED is blocked-with-handoff, not
"ratified with violations noted."

**The problem it names.** The validator currently has no consumers. Nothing
requires it to have run. It exists as a script someone may remember to
execute — and the evidence for that is three sessions of it not running:

1. **07-14.** The validator was built. Nothing was wired to it. The
   checkpoint recorded the gap in words — *"a contract is only a contract if
   it is the only door... agreed in-session, and then not filed"* — and filed
   nothing.
2. **07-14b.** The validator ran ONLY because the operator carried real output
   to a ctx that happened to remember the validator existed. No rule produced
   that. If that ctx had not remembered, the output would have moved on
   ungraded and the 33 validator bugs would still be undiscovered.
3. **07-15.** Two discovery runs completed and shipped as `ratified` this
   session and the previous one. **Neither was gated on the validator.** The
   rappers-handbook run's terminal state is `ratified` and the validator has
   never touched it. Its shape was checked by hand, by a ctx, in a chat.

**The evidence class is unusual and worth naming.** P11's three points are all
NON-occurrences — the failure it predicts has not yet cost anything, because
a human kept catching it. **That is exactly the shape of a hazard that looks
retired until the human is busy.** Do not read the absence of damage as
evidence the door is closed.

**The critical limit, recorded so ratification is honest.** The validator
checks SHAPE, never TRUTH. P-5 (provenance integrity) is unbuildable as a
lint: a fabricated `[operator-decided]` on a ruling no operator made is
perfectly well-formed and passes clean. **Making the validator the only door
does NOT make a green envelope mean a good run** — 07-14b's whole lesson. If
P11 is ratified, this paragraph rides with it, or the door becomes a rubber
stamp with authority.

**Strongest counter.** The validator failed on its first contact with real
output and at least two-thirds of what it reported were its own bugs. Making
a demonstrably buggy grader mandatory would have blocked the project's first
successful run on 49 violations, ~33 of them fictional. **A mandatory door
with a broken lock is worse than no door** — it converts a working run into a
blocked one and teaches operators to route around the gate, which is the
exact habit the rule exists to prevent.

**Response, and it constrains the ratification:** the counter is correct and
P11 must NOT be ratified before the validator's fixture rebuild (P12's
subject). Sequencing: rebuild the fixtures against pinned v2.6, run them
against BOTH existing real outputs (loeliger-til and rappers-handbook), and
only then close the door. **P11 ratified early is a foot-gun; P11 ratified
after two real runs pass is a contract.**

**Confidence: MEDIUM.** High on the diagnosis — the door is demonstrably open
and only luck has closed it three times. Medium on timing: it is the only
proposal in this register whose ratification has a hard prerequisite.

## P12 — A self-authored suite is provisional until real output tests it

**Status: MINTED 2026-07-15. Owed since 2026-07-14b, which named it as that
session's only clerical failure and did not file it.**

**The rule.** A test suite authored by the same ctx that authored the code it
tests is **PROVISIONAL**. Its green proves self-consistency, never
conformance. It becomes evidence only when real output — output the ctx did
not author — either contradicts it or fails to. Until then, a green self-test
is reported as *"green against fixtures I wrote"*, never as *"tested"*.

**Evidence — 2 points.**

1. **07-14b, the origin (lexicon W12).** The discovery validator shipped at
   **19/19 GREEN**, operator-run, against 1 valid + 18 negative fixtures. On
   first contact with real output it reported **49 violations, at least 33 of
   them its own bugs.** Root cause: the same ctx authored the spec-reading,
   the code, AND the fixtures — so the fixtures inherited the misreading and
   green was **structurally incapable** of detecting it. Green meant *the code
   agrees with me*. **This was predicted verbatim by the 07-14 checkpoint
   ("green means the code implements the spec — it does not mean the spec
   matches reality") and the ctx built the fixtures anyway and the failure
   happened anyway.**
2. **07-15, corroboration from the opposite direction.** v2.6 was authored by
   a ctx that had read the loeliger-til output; every fix was cut against that
   one source. Rather than self-test, it was run by a SEPARATE ctx against a
   DIFFERENT source (rappers-handbook, born_digital). All three fixes held
   under that separation — **and the separated run immediately found a defect
   the author could not have seen**: the ≤50 bound keys on container class,
   but the axis that blew it (`ingest:` at 12 lines vs. the assumed ~3) is
   expensive because of **how the source teaches** (a font-level signal), not
   what container it ships in. **A self-test would have scored 3/3 and learned
   nothing.** Separation is what produced the finding.

**Why it is primer-level, not a lexicon wart.** W12 already records the
instance. What the primer needs is the **standing consequence**: it changes
what a session may CLAIM at close and what a boot manifest may pin as a
receipt. The scorer's 18/18 is the identical construction and carries the
identical cell, unopened only because it has never met real output — and it
is pinned in every manifest as though it were evidence.

**Concrete obligations if ratified:**
- A ctx reports self-authored greens as *provisional*, with the authorship
  named. `topology_scorer_v2.py selftest → 18 PASS` is a receipt that the file
  is current; it is NOT a receipt that the scorer grades correctly.
- Where separation is available at no cost, **prefer it**: the ctx that mints
  fixtures should not be the ctx that scores the run (already law inside the
  topology kit; P12 generalizes it).
- A suite whose subject spec has changed is **KNOWN-STALE** until rebuilt
  (the validator's is, right now, and its rebuild is owed with v2.6).

**Strongest counter.** Taken literally this devalues nearly every green the
project has, including greens that are almost certainly fine, and it makes
testing more expensive by demanding a second ctx for work one ctx could do.
**Rebuttal:** it does not forbid the suite or the green — it forbids the
CLAIM. Write the fixtures, run them, report them honestly as self-authored.
The cost is one adjective. And 07-14b priced the alternative: a grader that
was confidently green certified nothing and cost a session to debug.

**Confidence: HIGH.** The mechanism is demonstrated in both directions —
self-authorship concealed a defect (07-14b), separation revealed one (07-15).

## How to use this register at a close

1. If you built on a hand-applied protection, **increment its evidence count
   and add the session ID.** One line. Do not restate the proposal.
2. If a session surfaced a NEW recurring failure, add a proposal with 1
   evidence point, drafted, with its strongest counter stated.
3. If a proposal was **argued against**, record the argument. A proposal that
   accumulates counters is as informative as one that accumulates evidence.
4. **Do not bump the primer from this file without an operator ruling.** That
   is the whole point of the file.
5. When the operator ratifies a batch, the ratifying session issues
   `workshop_primer_v5.md`, strikes the ratified proposals from this register,
   and re-files it as `primer_amendment_proposals_v2.md` with the survivors.
